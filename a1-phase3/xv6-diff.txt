+Part D: Design Document for xv6 User Program (find)
+====================================================
+
+Overview:
+---------
+The `find` program recursively searches a directory tree for files with a specific name, printing the full path of each match. It mimics the basic functionality of the UNIX `find` command, implemented for the xv6 operating system.
+
+Design Steps:
+-------------
+1. **Argument Parsing**
+   - The program expects two arguments: the starting directory path and the target filename.
+   - If the arguments are incorrect, it prints a usage message and exits.
+
+2. **Recursive Search Function (`find`)**
+   - The main logic is implemented in a recursive function `find(path, target)`.
+   - For each call:
+     - Opens the given path.
+     - Uses `fstat` to determine if the path is a file or directory.
+
+3. **File Handling**
+   - If the path is a file:
+     - Extracts the filename from the path.
+     - Compares it to the target name using `strcmp`.
+     - If it matches, prints the full path.
+
+4. **Directory Handling and Recursion**
+   - If the path is a directory:
+     - Reads each entry using `read` and `struct dirent`.
+     - Skips entries with `inum == 0` (empty slots).
+     - Skips "." and ".." to avoid infinite recursion.
+     - Builds the full path for each entry by appending the entry name to the current path.
+     - Recursively calls `find` on each entry.
+
+5. **Path Construction**
+   - Uses a buffer to construct new paths for recursive calls.
+   - Ensures buffer size is sufficient to avoid overflow.
+
+6. **Output**
+   - Only prints the full path of files that match the target filename.
+   - Output format matches assignment requirements (e.g., ./b, ./a/b, ./a/aa/b).
+
+7. **Error Handling**
+   - Handles errors for opening and stating files/directories.
+   - Skips entries or prints error messages as appropriate.
+
+References:
+-----------
+- Used `ls.c` from xv6 as a reference for directory reading and path construction.
+- Used C string functions (`strcpy`, `strlen`, `strcmp`, etc.) for path and name handling.
+
+Testing:
+--------
+- Tested with nested directories and files to ensure correct recursive search and output.
+- Verified output matches assignment requirements.
+
+Notes:
+------
+- The program is designed to be simple and robust, following xv6 conventions and constraints.
+- All code is commented for clarity and maintainability.

+Part D BONUS: Design Document for xv6 User Program (primes)
+==========================================================
+
+Overview:
+---------
+The `primes` program implements a concurrent prime sieve using pipes and processes, inspired by Doug McIlroy's Unix pipeline concept. It finds all prime numbers between 2 and 35 by setting up a chain of processes, each responsible for filtering multiples of a discovered prime.
+
+Design Steps:
+-------------
+1. **Pipeline Initialization**
+   - The main process creates a pipe and forks a child.
+   - The parent writes numbers 2 through 35 into the pipe, then waits for the child to finish.
+   - The child starts the sieve process by reading from the pipe.
+
+2. **Recursive Sieve Function**
+   - The sieve function reads the first number from its input pipe; this is the next prime.
+   - It prints the prime.
+   - It creates a new pipe and forks a child for the next sieve stage.
+   - The parent reads remaining numbers, filters out multiples of its prime, and writes the rest to the new pipe.
+   - The child recursively calls the sieve function on the new pipe.
+   - The recursion terminates when there are no more numbers to read from the pipe.
+
+3. **Process and Pipe Management**
+   - Each sieve stage is a separate process, connected by pipes.
+   - File descriptors are closed as soon as they are no longer needed to avoid resource exhaustion.
+   - The parent waits for its child to finish before exiting, ensuring all processes terminate cleanly.
+
+4. **Output**
+   - Each process prints its prime in the format: `prime <number>`.
+   - The output matches the assignment requirements, listing all primes between 2 and 35.
+
+5. **Error Handling and Termination**
+   - The base case for recursion is when the pipe is empty; the process closes its file descriptor and returns.
+   - All processes exit cleanly after their work is done.
+
+References:
+-----------
+- The design is based on the classic Unix pipeline prime sieve, as described in MIT's xv6 assignment and Doug McIlroy's original concept.
+- Used xv6 system calls: `pipe`, `fork`, `read`, `write`, `close`, `wait`, and `exit`.
+
+Testing:
+--------
+- Verified that the output matches the expected list of primes.
+- Checked that all processes terminate and no file descriptor/resource leaks occur.
+
+Notes:
+------
+- The recursive structure is elegant and matches the conceptual model of a pipeline of filters.
+- An explicit return in the base case is used to help the compiler recognize recursion termination.

+Part D: Test Plan for xv6 User Program (find)
+============================================
+
+Objective:
+----------
+Verify that the find program correctly searches for files with a given name in a directory tree and prints their full paths.
+
+Test Cases:
+-----------
+1. **Basic file search**
+   - Create a file named 'b' in the root directory.
+   - Run: find . b
+   - Expected output: ./b
+
+2. **Search in subdirectory**
+   - Create directory 'a', then file 'a/b'.
+   - Run: find . b
+   - Expected output: ./a/b
+
+3. **Search in nested subdirectory**
+   - Create directory 'a/aa', then file 'a/aa/b'.
+   - Run: find . b
+   - Expected output: ./a/aa/b
+
+4. **Multiple matches**
+   - All above files exist.
+   - Run: find . b
+   - Expected output:
+     ./b
+     ./a/b
+     ./a/aa/b
+
+5. **No matches**
+   - Run: find . z
+   - Expected output: (no output)
+
+6. **Search for a directory name**
+   - Run: find . a
+   - Expected output: (no output, since only files are matched)
+
+7. **Invalid arguments**
+   - Run: find .
+   - Expected output: Usage: find <path> <filename>
+
+8. **Path too long**
+   - Create deeply nested directories and file with long path.
+   - Run: find . b
+   - Expected output: Should not crash; may skip long paths.
+
+Instructions:
+-------------
+- Use xv6 shell commands (echo, mkdir, etc.) to set up test files and directories.
+- Run the find program with various arguments as described above.
+- Record the output for each test case in the test results file.

+# Part E: System Call Tracing Design Document
+
+## Overview
+This part implements a system call tracing feature in xv6. The trace syscall allows a user process to enable tracing for specific syscalls using a bitmask. When enabled, the kernel prints a trace line for each matching syscall, showing the process ID, syscall name, and return value.
+
+## Design Decisions
+- **Trace Mask:** Each process has a `trace_mask` field in its `struct proc`. The mask is set via the `trace` syscall and inherited on fork.
+- **Syscall Table:** The kernel maintains arrays mapping syscall numbers to handler functions and names. The mask uses `1 << syscall_number` to enable tracing for specific syscalls.
+- **Tracing Logic:** In the syscall handler, after executing a syscall, the kernel checks if tracing is enabled for that syscall and prints a trace line if so.
+- **User Program:** The `trace` user program parses the mask and command, calls the `trace` syscall, and execs the target program. It does not print extra output.
+
+## Implementation Steps
+1. Add `trace_mask` to `struct proc` in `proc.h` and ensure it is inherited in `proc.c` (fork).
+2. Implement `sys_trace` in `sysproc.c` to set the mask.
+3. Add `SYS_trace` to `syscall.h`, `user.h`, and `usys.pl`.
+4. Add `sys_trace` to the syscall table and prototypes in `syscall.c`.
+5. Update the syscall handler in `syscall.c` to print trace output if enabled.
+6. Implement the `trace` user program to set the mask and exec the target command.
+7. Update the Makefile to include `trace` in UPROGS.
+
+## Edge Cases
+- If a syscall number is not mapped, print the syscall number instead of the name.
+- The mask should only affect the calling process and its children.
+- The user program should handle incorrect arguments gracefully.
+

+/* Recursive sieve function: reads from read_fd, prints first prime, sets up next sieve */
+void sieve(int read_fd)
+{
+    int prime;
+    /* Read the first number (prime) from the pipe */
+    if(read(read_fd, &prime, sizeof(prime)) <= 0)
+    {
+        close(read_fd);
+        return; /* Explicit return to terminate recursion */
+    }
+    printf("prime %d\n", prime);
+
+    int p[2];
+    pipe(p);
+    int pid = fork();
+    if(pid == 0)
+    {
+        /* Child: next sieve stage */
+        close(p[1]);
+        sieve(p[0]);
+        exit(0);
+    }
+    else
+    {
+        /* Parent: filter and pass non-multiples to next pipe */
+        close(p[0]);
+        int num;
+        while(read(read_fd, &num, sizeof(num)) > 0)
+        {
+            if(num % prime != 0)
+            {
+                write(p[1], &num, sizeof(num));
+            }
+        }
+        close(read_fd);
+        close(p[1]);
+        wait(0);
+        exit(0);
+    }
